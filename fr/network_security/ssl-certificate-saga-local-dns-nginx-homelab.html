La Saga des Certificats SSL : DNS Local, Nginx et Mon Homelab

2025-10-26
8 min de lecture
Sécurité Réseau

La Saga Certificats SSL : DNS Local, Nginx et Mon Homelab

Objectif du Projet

Configurer un environnement local sécurisé avec : - Domaine public olyhome.site (acheté chez Hostinger) - Serveur DNS Local (Bind9) pour gérer les sous-domaines internes - Nginx Proxy Manager (NPM) pour le reverse proxy - Sous-domaines locaux sécurisés par HTTPS (ex : proxmox.olyhome.site, n8n.olyhome.site) - Sans exposer les services à Internet

Aperçu de l'Architecture

Local Network
    ↓
Bind9 (Local DNS)
├── proxmox.olyhome.site → 192.168.11.100
├── n8n.olyhome.site → 192.168.11.100
└── db.olyhome.site → 192.168.11.100
    ↓
Nginx Proxy Manager (192.168.11.100)
├── proxmox.olyhome.site → https://192.168.11.254:8006
├── n8n.olyhome.site → http://192.168.1.20:5678
└── db.olyhome.site → http://192.168.1.30:5432

Étapes Réalisées

Étape 1 : Configuration de Bind9 (DNS Local)

Pourquoi Bind9 ?

Cas d'utilisation : - Contrôler la résolution DNS pour le domaine local - Outrepasser le DNS public avec des IPs locales - Gérer le routage des sous-domaines internes - Ne pas dépendre des fournisseurs DNS externes

Installation

# Install Bind9
sudo apt update
sudo apt install bind9 bind9-utils -y

# Create zone directory
sudo mkdir -p /etc/bind/zones

Configuration de la Zone

# Edit named.conf.local
sudo nano /etc/bind/named.conf.local
// Zone for olyhome.site
zone "olyhome.site" {
    type master;
    file "/etc/bind/zones/db.olyhome.site";
};

Création du Fichier de Zone

# Create zone file
sudo nano /etc/bind/zones/db.olyhome.site
$TTL    604800
@       IN      SOA     ns1.olyhome.site. admin.olyhome.site. (
                        2025102601      ; Serial
                        604800          ; Refresh
                        86400           ; Retry
                        2419200         ; Expire
                        604800 )        ; Negative Cache TTL
;
@       IN      NS      ns1.olyhome.site.
ns1     IN      A       192.168.11.100

; Proxy server (Nginx Proxy Manager)
@       IN      A       192.168.11.100

; Service subdomains
proxmox IN      A       192.168.11.100
n8n     IN      A       192.168.11.100
db      IN      A       192.168.11.100
nginx   IN      A       192.168.11.100

Redémarrage et Test

# Check configuration
sudo named-checkconf
sudo named-checkzone olyhome.site /etc/bind/zones/db.olyhome.site

# Restart Bind9
sudo systemctl restart bind9

# Test DNS resolution
dig proxmox.olyhome.site @localhost

# Attendu : La section de réponse affiche 192.168.11.100

Étape 2 : Déploiement de Bind9 dans Docker

Pourquoi Docker ?

  • Isolation du système hôte
  • Sauvegarde et restauration faciles
  • Configuration portable
  • Pas de conflit avec le résolveur système

Docker Compose Configuration

version: '3.8'

services:
  bind9:
    image: ubuntu/bind9:latest
    container_name: bind9-dns
    restart: unless-stopped
    ports:
      - "53:53/tcp"
      - "53:53/udp"
      - "953:953/tcp"  # rndc
    volumes:
      - ./config:/etc/bind
      - ./zones:/var/lib/bind
      - ./cache:/var/cache/bind
    environment:
      - TZ=Europe/Paris
      - BIND9_USER=bind
    networks:
      - dns_network

networks:
  dns_network:
    driver: bridge

Résolution des Conflits

Problème : Le port 53 est déjà utilisé par systemd-resolved

# Check what's using port 53
sudo ss -tlnp | grep :53

# Disable systemd-resolved
sudo systemctl disable systemd-resolved
sudo systemctl stop systemd-resolved

# Remove resolv.conf symlink
sudo rm /etc/resolv.conf

# Create static resolv.conf
echo "nameserver 127.0.0.1" | sudo tee /etc/resolv.conf

# Start Bind9 container
docker-compose up -d

# Verify port 53 is bound
docker logs bind9-dns

Étape 3 : Configuration de Nginx Proxy Manager

Déploiement

version: '3'
services:
  nginx-proxy-manager:
    image: 'jc21/nginx-proxy-manager:latest'
    container_name: nginx-proxy-manager
    restart: unless-stopped
    ports:
      - '80:80'
      - '81:81'
      - '443:443'
    volumes:
      - ./data:/data
      - ./letsencrypt:/etc/letsencrypt
    networks:
      - proxy_network

networks:
  proxy_network:
    driver: bridge
# Deploy NPM
docker-compose up -d

# Access web interface
http://192.168.11.100:81

Étape 4 : Tentative de Certificat SSL

Objectif : Certificats SSL Valides pour les Domaines Locaux

Défi : Nginx Proxy Manager ne prend pas en charge Hostinger pour les défis DNS automatisés.

Méthode : Certbot Manuel avec Défi DNS

# Install certbot
sudo apt install certbot -y

# Request certificate manually
sudo certbot certonly \
  --manual \
  --preferred-challenges dns \
  -d olyhome.site \
  -d '*.olyhome.site'

Processus : 1. Certbot fournit la valeur de l'enregistrement TXT 2. Ajouter l'enregistrement TXT au DNS de Hostinger : _acme-challenge.olyhome.site TXT "generated-value-here" 3. Attendre la propagation DNS (2-5 minutes) 4. Appuyer sur Entrée dans certbot 5. Certificat généré avec succès

Emplacement du Certificat :

/etc/letsencrypt/live/olyhome.site/fullchain.pem
/etc/letsencrypt/live/olyhome.site/privkey.pem

Étape 5 : Problème d'Importation NPM

Problème Rencontré

Erreur : NPM refuse d'importer privkey.pem

Message : « La clé privée est chiffrée/possède une phrase de passe »

Solutions Tentées

Solution 1 : Supprimer la Phrase de Passe avec OpenSSL RSA

# Tenter de supprimer la phrase de passe
openssl rsa -in privkey.pem -out privkey-nopass.pem

# ERREUR: "Not an RSA key"
# Let's Encrypt utilise des clés ECDSA, pas RSA !

Solution 2 : Convertir la Clé EC

# Vérifier le type de clé
openssl pkey -in privkey.pem -text -noout

# Convertir la clé EC (supprimer la phrase de passe si présente)
openssl ec -in privkey.pem -out privkey-nopass.pem

# Succès : privkey-nopass.pem créé

Solution 3 : Importer dans NPM

Interface Utilisateur NPM :
Certificats SSL → Ajouter un Certificat SSL → Personnalisé
Clé de Certificat : Télécharger privkey-nopass.pem
Certificat : Télécharger fullchain.pem
Certificats Intermédiaires : (laisser vide)

Résultat : L'importation échoue toujours avec une erreur générique

Problème Actuel

Statut : Impossible d'importer un certificat Let’s Encrypt valide dans Nginx Proxy Manager malgré la suppression de tout chiffrement de la clé privée.

Erreur : NPM affiche un message générique « Failed to save certificate » sans détails spécifiques.

Dépannage Effectué

1. Vérifier le Format du Certificat

# Vérifier fullchain.pem
openssl x509 -in fullchain.pem -text -noout

# Vérifier qu'il contient :
# - Certificat
# - Subject Alternative Names (SAN): *.olyhome.site
# - Issuer: Let's Encrypt Authority

# Vérifier privkey.pem
openssl pkey -in privkey-nopass.pem -check

# Output: "EC Key valid"

2. Vérifier les Caractères Supplémentaires

# S'assurer qu'il n'y a pas d'espace blanc/caractères supplémentaires
head -n 1 fullchain.pem
# Devrait être exactement : -----BEGIN CERTIFICATE-----

tail -n 1 fullchain.pem
# Devrait être exactement : -----END CERTIFICATE-----

3. Essayer Différentes Méthodes d'Importation

Méthode A : Via l'IU de NPM ❌ Échec

Méthode B : Copie Directe de Fichiers

# Copier vers le conteneur NPM
docker cp fullchain.pem nginx-proxy-manager:/data/custom_ssl/
docker cp privkey-nopass.pem nginx-proxy-manager:/data/custom_ssl/

# Tenter l'importation via l'IU en utilisant les chemins du conteneur
# Échoue toujours

Méthode C : Configuration Manuelle de Nginx

# Accéder au conteneur NPM
docker exec -it nginx-proxy-manager bash

# Modifier directement la configuration de l'hôte proxy
nano /data/nginx/proxy_host/1.conf
server {
    listen 443 ssl http2;
    server_name proxmox.olyhome.site;

    ssl_certificate /data/custom_ssl/fullchain.pem;
    ssl_certificate_key /data/custom_ssl/privkey-nopass.pem;

    location / {
        proxy_pass https://192.168.11.254:8006;
        # ... rest of proxy config
    }
}

Solutions Alternatives Envisagées

Option 1 : Utiliser un Certificat Auto-Signé

# Générer un certificat auto-signé au format NPM
openssl req -x509 -nodes -days 365 -newkey rsa:2048 \
  -keyout selfsigned-key.pem \
  -out selfsigned-cert.pem \
  -subj "/CN=*.olyhome.site"

# Importer dans NPM (fonctionne, mais avec avertissements du navigateur)

Option 2 : Use mkcert

# Installer mkcert
wget https://github.com/FiloSottile/mkcert/releases/download/v1.4.4/mkcert-v1.4.4-linux-amd64
sudo mv mkcert-v1.4.4-linux-amd64 /usr/local/bin/mkcert
sudo chmod +x /usr/local/bin/mkcert

# Installer l'AC locale
mkcert -install

# Générer un certificat
mkcert "*.olyhome.site" olyhome.site

# Importer dans NPM (devrait fonctionner, installation de l'AC cliente requise)

Option 3 : Reverse Proxy Nginx Manuel

# Ignorer complètement NPM
# Installer nginx directement
sudo apt install nginx -y

# Configurer les hôtes virtuels manuellement
# Plus de contrôle, pas de restrictions d'interface graphique

Option 4 : Caddy Server

# Install Caddy
sudo apt install -y debian-keyring debian-archive-keyring apt-transport-https
curl -1sLf 'https://dl.cloudsmith.io/public/caddy/stable/gpg.key' | sudo gpg --dearmor -o /usr/share/keyrings/caddy-stable-archive-keyring.gpg

# Caddy gère automatiquement Let's Encrypt
# Configuration simple :
proxmox.olyhome.site {
    reverse_proxy https://192.168.11.254:8006 {
        transport http {
            tls_insecure_skip_verify
        }
    }
}

Leçons Apprises

1. Limitations de NPM

  • Ne prend pas en charge tous les fournisseurs DNS pour les défis automatisés
  • L'importation de certificats personnalisés peut être capricieuse
  • Rapport d'erreurs limité pour le dépannage
  • Les reverse proxies alternatifs peuvent être plus flexibles

2. Le Format du Certificat Compte

  • Let’s Encrypt utilise des clés ECDSA (pas RSA)
  • Utiliser openssl ec pour ECDSA, pas openssl rsa
  • Le format PEM doit être exact (pas d'espace blanc supplémentaire)

3. Le DNS Local avec Domaine Public Fonctionne

  • Bind9 outrepasse avec succès le DNS public
  • Les clients résolvent les IPs internes pour le domaine public
  • Permet le HTTPS pour les services locaux avec des certificats valides

4. Le Défi DNS est Puissant

  • Fonctionne sans ouvrir les ports 80/443
  • Permet les certificats wildcard
  • Nécessite des mises à jour manuelles des enregistrements DNS pour les fournisseurs non pris en charge par les plugins certbot

Prochaines Étapes

    • Mieux adapté aux environnements conteneurisés
  3. Implémenter la surveillance :
    • Alertes d'expiration de certificat

Conclusion

Bien que l'importation du certificat SSL dans NPM reste non résolue, le projet a démontré avec succès : - La configuration DNS locale avec Bind9 - Le DNS Split-horizon (IPs internes pour le domaine public) - La configuration de l'hôte proxy dans Nginx Proxy Manager - La génération manuelle de certificat Let’s Encrypt avec défi DNS

L'incapacité d'importer des certificats dans NPM souligne l'intérêt d'explorer des solutions de reverse proxy alternatives comme Caddy ou Traefik, qui offrent une meilleure automatisation et moins de restrictions d'importation.

Point Clé : Parfois, la limitation d'un outil n'est pas un échec, c'est une opportunité de découvrir de meilleures solutions.
