Dépannage de l'accès à l'interface Web pfSense : Un guide étape par étape

Objectif

Accéder à l'interface web pfSense via l'interface WAN (ex: 192.168.11.253) depuis un poste client (ex: 192.168.11.250). C'est une exigence courante lorsque pfSense est déployé en tant que VM ou lorsque vous avez besoin d'un accès d'administration depuis un segment réseau spécifique.

Aperçu du Problème

Par défaut, pfSense bloque l'accès à son interface web depuis le côté WAN pour des raisons de sécurité. Ce tutoriel montre comment activer cet accès en toute sécurité lorsque cela est nécessaire.

Symptômes

Indicateurs d'Erreur Courants : - Impossible d'accéder à https://192.168.11.253 depuis le client - Délai de connexion (Connection timeout) lors de la navigation vers l'IP WAN de pfSense - Les logs du pare-feu montrent des connexions bloquées

Exemple d'Entrée de Log :

block in on em0 from 192.168.11.250 to 192.168.11.253 port 443

Avertissement de Sécurité

⚠️ Important : Autoriser l'accès WAN à l'interface GUI de pfSense représente un risque de sécurité. N'activez cette fonctionnalité que temporairement pour le dépannage ou lorsque cela est absolument nécessaire. Restreignez toujours l'accès à des IPs de confiance spécifiques.

Étapes de Dépannage

Étape 1 : Vérifier que le service GUI est en cours d'exécution

Accédez à la console pfSense (via accès physique, IPMI ou console Proxmox) :

# pfSense console menu
Option 11) Restart webConfigurator

# Wait for service to restart
# Verify it's listening

Sortie Attendue :

Restarting webConfigurator... done.

Étape 2 : Vérifier la configuration de l'interface WAN

# Option 8) Shell
ip a

# Or from pfSense console menu:
# Option 2) Set interface(s) IP address

Vérifiez : - L'interface WAN (em0, vtnet0, etc.) a la bonne IP - L'IP doit être 192.168.11.253 (ou votre IP WAN attendue) - Le statut de l'interface indique « UP »

Étape 3 : Tester avec le Pare-feu Temporairement Désactivé

⚠️ AVERTISSEMENT : Uniquement à des fins de test !

# Disable pfSense firewall (from console shell)
pfctl -d

# Test access from client
# Open browser: https://192.168.11.253

# Re-enable firewall immediately after test
pfctl -e

Si cela fonctionne : Le problème vient des règles de pare-feu (passez à l'Étape 4) Si cela ne fonctionne pas : Le problème vient du service web ou de la connectivité réseau

Étape 4 : Ajouter une Règle de Pare-feu Temporaire via le Shell

# Access pfSense shell (Option 8)

# Add temporary pass rule for your IP
echo "pass in quick on em0 proto tcp from 192.168.11.250 to 192.168.11.253 port 443 keep state" | pfctl -f -

# Replace 'em0' with your actual WAN interface name
# Replace IP addresses with your actual values

Testez à nouveau l'accès : https://192.168.11.253

Si réussi : La règle de pare-feu est la solution, rendez-la persistante (Étape 5)

Étape 5 : Créer une Règle de Pare-feu Persistante via l'Interface Web

Une fois que vous pouvez accéder à l'interface web :

  1. Naviguez vers les Règles de Pare-feu :

    Firewall → Rules → WAN → Add (bouton flèche vers le haut)

  2. Configurez la Règle :

    • Action : Pass (Autoriser)
    • Interface : WAN
    • Famille d'Adresses : IPv4
    • Protocole : TCP
    • Source :
      • Type : Single host or alias (Hôte unique ou alias)
      • Adresse : 192.168.11.250 (votre PC d'administration)
    • Destination :
      • Type : WAN address (Adresse WAN)
      • Port : HTTPS (443)
    • Description : Autoriser l'accès GUI depuis le PC d'administration
    • Log : ✓ (optionnel, pour la surveillance)

  3. Options Avancées (Recommandées) :

    • Gateway : default
    • Schedule : None (ou créez un calendrier pour un accès basé sur l'heure)

  4. Sauvegarder et Appliquer les Changements

Étape 6 : Vérifier que le Service Web est à l'écoute

# From pfSense shell
sockstat -4 | grep :443

# Expected output:
# nginx    12345 root    6u  IPv4 0x... TCP *:443 (LISTEN)

Si le service n'est pas à l'écoute :

# Restart web service
/etc/rc.restart_webgui

# Or from console menu
Option 11) Restart webConfigurator

Identifiants par Défaut

Connexion par Défaut pfSense : - Nom d'utilisateur : admin - Mot de passe : pfsense

Si les identifiants sont oubliés :

# From pfSense console
Option 3) Reset webConfigurator password

# Follow prompts to reset admin password

Meilleures Pratiques de Sécurité

1. Restreindre l'Accès par Adresse IP

N'autorisez que des IPs de confiance spécifiques à accéder à l'interface WAN :

Firewall → Rules → WAN
Source: Single host → 192.168.11.250
PAS : Any (non sécurisé !)

2. Utiliser l'Authentification par Clé SSH

Au lieu de l'accès web, envisagez SSH :

# From client
ssh admin@192.168.11.253

# pfSense console appears
# More secure than web interface

3. Mettre en Œuvre l'Authentification Multi-Facteurs

System → User Manager → Users → admin
→ Edit → "2FA" section
→ Enable Google Authenticator or other TOTP

4. Changer le Port par Défaut

System → Advanced → Admin Access
HTTPS Port: 8443 (au lieu de 443)

# Accès via : https://192.168.11.253:8443

5. Utiliser des Alias pour l'Administration

Créez un alias de pare-feu pour les IPs d'administration :

Firewall → Aliases → IP
Name: mgmt_workstations
Type: Host(s)
IP: 192.168.11.250
Description: PCs d'administration autorisés

# À utiliser dans les règles de pare-feu au lieu des IPs individuelles

Dépannage Avancé

Problème 1 : Règle Ajoutée mais Toujours Bloquée

Vérifiez l'ordre des règles : - Les règles sont traitées de haut en bas - Assurez-vous que la règle d'autorisation (pass rule) est AU-DESSUS de toute règle de blocage - Déplacez la règle vers le haut si nécessaire

Vérifiez que la règle est sur la bonne interface : - Les règles WAN s'appliquent au trafic entrant dans l'interface WAN - Les règles LAN s'appliquent au trafic entrant dans l'interface LAN

Problème 2 : Avertissements de Certificat HTTPS

Problème : Le navigateur affiche un avertissement de sécurité

Solution :

System → Cert Manager → Certificates
→ Add (créer un certificat auto-signé)

System → Advanced → Admin Access
→ SSL Certificate: Sélectionnez votre certificat

Problème 3 : Délai de Connexion (Non Bloqué)

Causes possibles : 1. Problèmes MTU 2. Routage asymétrique 3. Table d'état pleine

Solutions :

# Check state table
pfctl -s state | grep 192.168.11.250

# Increase state table size
System → Advanced → Firewall & NAT
Maximum table entries: 200000 (default: 400000)

Problème 4 : L'Accès Fonctionne Puis S'arrête

Vérifiez le délai d'expiration de session :

System → Advanced → Admin Access
Session timeout: 240 (par défaut, en minutes)

# Augmentez si nécessaire pour de longues sessions d'administration

Surveillance et Journalisation

Activer la Journalisation pour l'Accès WAN

Firewall → Rules → WAN → Edit rule
Log packets matched: ✓ (Journaliser les paquets correspondants)

# Afficher les logs :
Status → System Logs → Firewall
Filter by: WAN, 443, votre IP

Créer une Alerte pour l'Accès WAN

System → Advanced → Notifications
→ Add notification (Ajouter notification)
Type: Email
Event: Firewall block/pass on WAN

# Être notifié des tentatives d'accès

Méthodes d'Accès Alternatives

Option 1 : Accès VPN

Plus sécurisé que l'accès WAN direct :

VPN → OpenVPN → Wizards
→ Configure OpenVPN server
→ Connect via VPN first (Connectez-vous d'abord via VPN)
→ Access GUI via LAN interface (more secure) (Accédez à l'interface GUI via l'interface LAN - plus sécurisé)

Option 2 : Cloudflare Tunnel

Accès Zero Trust sans exposer les ports :

Install cloudflared on pfSense
Create tunnel to pfSense GUI
Access via: https://pfsense.yourdomain.com
No firewall rules needed on WAN

Option 3 : Tunneling SSH

# Create SSH tunnel from client
ssh -L 8443:localhost:443 admin@192.168.11.253

# Access GUI via tunnel
https://localhost:8443

# More secure than direct web access

Liste de Contrôle Complète

Étapes de Diagnostic : - [ ] Vérifier que le service web pfSense est en cours d'exécution - [ ] Confirmer que l'interface WAN a la bonne IP - [ ] Tester avec le pare-feu temporairement désactivé - [ ] Vérifier les logs du pare-feu pour les connexions bloquées - [ ] Vérifier que le service est à l'écoute sur le port 443

Étapes de Sécurité : - [ ] Restreindre l'IP source au poste d'administration uniquement - [ ] Utiliser un mot de passe fort (pas celui par défaut) - [ ] Activer 2FA pour le compte admin - [ ] Envisager d'utiliser un port non standard - [ ] Activer la journalisation des tentatives d'accès - [ ] Examiner régulièrement les logs du pare-feu

Post-Configuration : - [ ] Tester l'accès depuis l'IP autorisée - [ ] Vérifier que l'accès est refusé depuis d'autres IPs - [ ] Documenter la règle de pare-feu dans le wiki de votre homelab - [ ] Définir un rappel dans le calendrier pour revoir les règles d'accès tous les mois

Notes Additionnelles

Pourquoi pfSense Bloque l'Accès WAN par Défaut

Raisonnement Sécuritaire : - L'interface WAN fait face à des réseaux non fiables (Internet) - L'interface web est une cible d'attaque de grande valeur - Le blocage par défaut (Default deny) empêche l'accès non autorisé - Force une approche d'autorisation explicite (allowlist)

Quand Autoriser l'Accès WAN

Cas d'Utilisation Légitimes : - pfSense déployé en tant que VM (WAN = réseau homelab) - Administration depuis un réseau admin isolé - Dépannage temporaire (désactiver après) - Environnement de laboratoire contrôlé

NON Recommandé : - pfSense WAN connecté directement à Internet - Aucune restriction IP (toute source) - Environnements de production sans 2FA

Conclusion

Accéder à l'interface web pfSense via WAN nécessite une configuration de pare-feu minutieuse. Suivez toujours le principe du moindre privilège : n'autorisez que des IPs de confiance spécifiques, activez la journalisation et envisagez des méthodes d'accès alternatives comme le VPN ou le tunneling SSH pour une sécurité renforcée.

Points Clés à Retenir : - Le blocage WAN par défaut est une fonctionnalité de sécurité intentionnelle - Utilisez des restrictions IP, jamais « toute source » (any source) - Activez 2FA et une authentification forte - Envisagez le VPN/tunnel pour une utilisation en production - Surveillez et journalisez toutes les tentatives d'accès WAN - Documentez votre configuration

Pour les déploiements en production, utilisez toujours des solutions VPN ou Zero Trust au lieu d'un accès WAN direct.
