Dépannage de l'accès à l'interface Web pfSense : Un guide étape par étape

Objectif

Accéder à l'interface Web pfSense via l'interface WAN (par exemple, 192.168.11.253) depuis un poste client (par exemple, 192.168.11.250). C'est une exigence courante lorsque pfSense est déployé en tant que VM ou lorsque vous avez besoin d'un accès de gestion depuis un segment réseau spécifique.

Aperçu du Problème

Par défaut, pfSense bloque l'accès à son interface Web depuis le côté WAN pour des raisons de sécurité. Ce tutoriel montre comment activer l'accès en toute sécurité lorsque cela est nécessaire.

Symptômes

Indicateurs d'Erreur Courants : - Impossible d'accéder à https://192.168.11.253 depuis le client - Délai d'expiration de la connexion (timeout) lors de la navigation vers l'IP WAN de pfSense - Les journaux du pare-feu montrent des connexions bloquées

Exemple d'Entrée de Journal :

block in on em0 from 192.168.11.250 to 192.168.11.253 port 443

Avertissement de Sécurité

⚠️ Important: Autoriser l'accès WAN à l'interface graphique de pfSense est un risque de sécurité. N'activez cette option que temporairement pour le dépannage ou lorsque cela est absolument nécessaire. Restreignez toujours l'accès à des adresses IP spécifiques et fiables.

Étapes de Dépannage

Étape 1 : Vérifier que le Service d'Interface Graphique est en Cours d'Exécution

Accédez à la console pfSense (via accès physique, IPMI, ou console Proxmox) :

# Menu de la console pfSense
Option 11) Restart webConfigurator

# Attendre le redémarrage du service
# Vérifier qu'il est en écoute

Sortie Attendue :

Restarting webConfigurator... done.

Étape 2 : Vérifier la Configuration de l'Interface WAN

# Option 8) Shell
ip a

# Ou depuis le menu de la console pfSense :
# Option 2) Set interface(s) IP address

Vérifier : - L'interface WAN (em0, vtnet0, etc.) a la bonne adresse IP - L'IP devrait être 192.168.11.253 (ou votre IP WAN attendue) - Le statut de l'interface indique « UP » (Actif)

Étape 3 : Tester avec le Pare-feu Temporairement Désactivé

⚠️ AVERTISSEMENT : Uniquement à des fins de test !

# Désactiver le pare-feu pfSense (depuis le shell de la console)
pfctl -d

# Tester l'accès depuis le client
# Open browser: https://192.168.11.253

# Réactiver le pare-feu immédiatement après le test
pfctl -e

Si cela fonctionne : Le problème est lié aux règles du pare-feu (passer à l'Étape 4) Si cela ne fonctionne pas : Le problème est lié au service web ou à la connectivité réseau

Étape 4 : Ajouter une Règle de Pare-feu Temporaire via le Shell

# Accéder au shell pfSense (Option 8)

# Ajouter une règle d'autorisation temporaire pour votre IP
echo "pass in quick on em0 proto tcp from 192.168.11.250 to 192.168.11.253 port 443 keep state" | pfctl -f -

# Remplacer 'em0' par le nom réel de votre interface WAN
# Remplacer les adresses IP par vos valeurs réelles

Tester l'accès à nouveau : https://192.168.11.253

Si succès : La règle de pare-feu est la solution, rendez-la persistante (Étape 5)

Étape 5 : Créer une Règle de Pare-feu Persistante via l'Interface Web

Une fois que vous pouvez accéder à l'interface web :

  1. Naviguer vers les Règles du Pare-feu :

    Firewall → Rules → WAN → Add (up arrow button)

  2. Configurer la Règle :

    • Action : Passer (Pass)
    • Interface: WAN
    • Address Family: IPv4
    • Protocol: TCP
    • Source :
      • Type : Hôte unique ou alias
      • Adresse : 192.168.11.250 (votre PC de gestion)
    • Destination :
      • Type : Adresse WAN
      • Port: HTTPS (443)
    • Description : Autoriser l'accès à l'interface graphique depuis le PC de gestion
    • Journalisation (Log) : ✓ (optionnel, pour la surveillance)

  3. Options Avancées (Recommandé) :

    • Gateway: default
    • Planification (Schedule) : Aucune (ou créer une planification pour un accès basé sur l'heure)

  4. Sauvegarder et Appliquer les Changements

Étape 6 : Vérifier que le Service Web est en Écoute

# From pfSense shell
sockstat -4 | grep :443

# Sortie attendue :
# nginx    12345 root    6u  IPv4 0x... TCP *:443 (LISTEN)

Si le service n'est pas en écoute :

# Redémarrer le service web
/etc/rc.restart_webgui

# Ou depuis le menu de la console
Option 11) Restart webConfigurator

Identifiants par Défaut

Identifiants par Défaut pfSense : - Username : admin - Mot de passe : pfsense

Si les identifiants sont oubliés :

# Depuis la console pfSense
Option 3) Reset webConfigurator password

# Suivre les instructions pour réinitialiser le mot de passe admin

Meilleures Pratiques de Sécurité

1. Restreindre l'Accès par Adresse IP

Autoriser uniquement des adresses IP fiables spécifiques à accéder à l'interface WAN :

Firewall → Rules → WAN
Source: Single host → 192.168.11.250
NON: Any (non sécurisé !)

2. Utiliser l'Authentification par Clé SSH

Au lieu de l'accès web, envisagez d'utiliser SSH :

# From client
ssh admin@192.168.11.253

# La console pfSense apparaît
# Plus sécurisé que l'interface web

3. Mettre en Œuvre l'Authentification Multi-Facteurs

System → User Manager → Users → admin
→ Modifier → "2FA" section
→ Activer Google Authenticator ou autre TOTP

4. Changer le Port par Défaut

System → Advanced → Admin Access
HTTPS Port: 8443 (instead of 443)

# Access via: https://192.168.11.253:8443

5. Utiliser des Alias pour la Gestion

Créer un alias de pare-feu pour les adresses IP de gestion :

Firewall → Aliases → IP
Nom: mgmt_workstations
Type: Host(s)
IP: 192.168.11.250
Description: PCs de gestion autorisés

# Utiliser dans les règles de pare-feu au lieu d'IP individuelles

Dépannage Avancé

Problème 1 : Règle Ajoutée mais Toujours Bloquée

Vérifier l'ordre des règles : - Les règles sont traitées de haut en bas - S'assurer que la règle d'autorisation (pass) est AU-DESSUS de toute règle de blocage - Déplacer la règle vers le haut si nécessaire

Vérifier que la règle est sur la bonne interface : - Les règles WAN s'appliquent au trafic entrant sur l'interface WAN - Les règles LAN s'appliquent au trafic entrant sur l'interface LAN

Problème 2 : Avertissements de Certificat HTTPS

Problème : Le navigateur affiche un avertissement de sécurité

Solution :

System → Cert Manager → Certificates
→ Ajouter (créer un certificat auto-signé)

System → Advanced → Admin Access
→ Certificat SSL : Sélectionner votre certificat

Problème 3 : Délai d'Expiration de la Connexion (Non Bloqué)

Causes possibles : 1. Problèmes de MTU 2. Routage asymétrique 3. Table d'état pleine

Solutions :

# Vérifier la table d'état
pfctl -s state | grep 192.168.11.250

# Augmenter la taille de la table d'état
System → Advanced → Firewall & NAT
Maximum table entries: 200000 (par défaut: 400000)

Problème 4 : L'Accès Fonctionne puis S'Arrête

Vérifier le délai d'expiration de la session :

System → Advanced → Admin Access
Délai d'expiration de la session: 240 (par défaut, en minutes)

# Augmenter si nécessaire pour les longues sessions d'administration

Surveillance et Journalisation

Activer la Journalisation pour l'Accès WAN

Firewall → Rules → WAN → Edit rule
Journaliser les paquets correspondants : ✓

# Afficher les journaux :
Status → System Logs → Firewall
Filtrer par: WAN, 443, votre IP

Créer une Alerte pour l'Accès WAN

System → Advanced → Notifications
→ Ajouter une notification
Type: Email
Event: Firewall block/pass on WAN

# Être notifié des tentatives d'accès

Méthodes d'Accès Alternatives

Option 1 : Accès VPN

Plus sécurisé que l'accès WAN direct :

VPN → OpenVPN → Wizards
→ Configure OpenVPN server
→ Se connecter via VPN d'abord
→ Accéder à l'interface graphique via l'interface LAN (plus sécurisé)

Option 2 : Tunnel Cloudflare

Accès Zero Trust sans exposer les ports :

Install cloudflared on pfSense
Créer un tunnel vers l'interface graphique pfSense
Access via: https://pfsense.yourdomain.com
Aucune règle de pare-feu nécessaire sur WAN

Option 3 : Tunneling SSH

# Créer un tunnel SSH depuis le client
ssh -L 8443:localhost:443 admin@192.168.11.253

# Accéder à l'interface graphique via le tunnel
https://localhost:8443

# Plus sécurisé que l'accès web direct

Liste de Contrôle Complète

Étapes de Diagnostic : - [ ] Vérifier que le service web pfSense est en cours d'exécution - [ ] Confirmer que l'interface WAN a la bonne IP - [ ] Tester avec le pare-feu temporairement désactivé - [ ] Vérifier les journaux du pare-feu pour les connexions bloquées - [ ] Vérifier que le service est en écoute sur le port 443

Étapes de Sécurité : - [ ] Restreindre l'IP source au poste de travail de gestion uniquement - [ ] Utiliser un mot de passe fort (pas celui par défaut) - [ ] Activer l'A2F (2FA) pour le compte admin - [ ] Envisager d'utiliser un port non standard - [ ] Activer la journalisation pour les tentatives d'accès - [ ] Examiner régulièrement les journaux du pare-feu

Post-Configuration : - [ ] Tester l'accès depuis l'IP autorisée - [ ] Vérifier que l'accès est refusé depuis d'autres IPs - [ ] Documenter la règle de pare-feu dans votre wiki homelab - [ ] Définir un rappel dans l'agenda pour réviser les règles d'accès mensuellement

Notes Supplémentaires

Pourquoi pfSense Bloque l'Accès WAN par Défaut

Raisonnement de Sécurité : - L'interface WAN fait face à des réseaux non fiables (Internet) - L'interface Web est une cible d'attaque de grande valeur - Le refus par défaut empêche l'accès non autorisé - Force une approche explicite par liste blanche (allowlist)

Quand Autoriser l'Accès WAN

Cas d'Usage Légitimes : - pfSense déployé en tant que VM (WAN = réseau homelab) - Gestion depuis un réseau d'administration isolé - Dépannage temporaire (désactiver après) - Environnement de laboratoire contrôlé

NON Recommandé : - pfSense WAN connecté directement à Internet - Aucune restriction d'IP (toute source) - Environnements de production sans A2F

Conclusion

Accéder à l'interface Web pfSense via le WAN nécessite une configuration rigoureuse du pare-feu. Suivez toujours le principe du moindre privilège : n'autorisez que des adresses IP fiables spécifiques, activez la journalisation et envisagez des méthodes d'accès alternatives comme le VPN ou le tunneling SSH pour une sécurité renforcée.

Points Clés : - Le blocage WAN par défaut est une fonctionnalité de sécurité intentionnelle - Utilisez des restrictions d'IP, jamais « toute source » (any source) - Activer l'A2F et une authentification forte - Envisager le VPN/tunnel pour une utilisation en production - Surveiller et journaliser toutes les tentatives d'accès WAN - Documenter votre configuration

Pour les déploiements en production, utilisez toujours des solutions VPN ou Zero Trust au lieu d'un accès WAN direct.
