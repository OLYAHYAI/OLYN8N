Sécuriser mes Workflows N8N : Plongée Profonde dans le DNS et le TLS

Objectif

Établir un accès fiable et chiffré à n8n (et aux services associés) soit publiquement (challenge Let’s Encrypt DNS-01) soit sur le réseau local (LAN) (Split-DNS + certificat local).

Pourquoi sécuriser n8n ?

n8n est une plateforme d'automatisation puissante qui gère souvent des informations d'identification sensibles, des clés API et des workflows critiques pour l'entreprise. Sécuriser l'accès avec un DNS et un TLS appropriés garantit :

  • Protection des données : Les connexions chiffrées empêchent l'interception des identifiants
  • Authentification : Les certificats vérifiés empêchent les attaques de l'homme du milieu (man-in-the-middle)
  • Conformité : Respecte les meilleures pratiques de sécurité pour l'automatisation des workflows
  • Configuration professionnelle : Démontre la maturité de l'infrastructure

Aperçu de l'Architecture

Internet/LAN Clients
       ↓
   pfSense (DNS + Firewall)
       ↓
Nginx Proxy Manager (TLS Termination)
       ↓
   n8n Docker Container (5678)

Option A : Domaine Public + Cloudflare + Challenge DNS (Let’s Encrypt)

Prérequis

  • Domaine public (peut utiliser des services gratuits comme Freenom)
  • Compte Cloudflare (le niveau gratuit est suffisant)
  • Nginx Proxy Manager déployé
  • n8n exécuté dans Docker

Étape 1 : Configuration du Domaine avec Cloudflare

Enregistrer un Domaine Gratuit (Facultatif)

# Visit Freenom.com
# Register domain: my-portfolio.tk (or similar)
# Note: Free domains may have limitations

Ajouter le Domaine à Cloudflare

  1. Créez un compte Cloudflare sur cloudflare.com
  2. Cliquez sur « Ajouter un site »
  3. Entrez votre nom de domaine
  4. Choisissez le plan Gratuit
  5. Copiez les serveurs de noms Cloudflare

Mettre à Jour les Serveurs de Noms chez le Registraire

  1. Connectez-vous à votre registraire de domaine (Freenom, Namecheap, etc.)

  2. Naviguez jusqu'à DNS Management (Gestion DNS)

  3. Remplacez les serveurs de noms existants par les serveurs de noms Cloudflare :

    ns1.cloudflare.com
ns2.cloudflare.com

  4. Attendez la propagation (cela peut prendre jusqu'à 24 heures)

Étape 2 : Générer un Jeton API Cloudflare

  1. Dans le tableau de bord Cloudflare, cliquez sur l'icône de profil
  2. Naviguez vers « Mon Profil » → « Jetons API »
  3. Cliquez sur « Créer un Jeton »
  4. Utilisez le modèle : « Modifier le DNS de la zone »
  5. Configurez les permissions :
    • Zone : Modification DNS (DNS Edit)
    • Inclure : Zone spécifique (votre domaine)
  6. Cliquez sur « Continuer vers le résumé » → « Créer le Jeton »
  7. Sauvegardez ce jeton en toute sécurité - vous ne le verrez qu'une seule fois !

Étape 3 : Configurer Nginx Proxy Manager

Créer l'Enregistrement DNS dans Cloudflare

  1. Dans Cloudflare, allez à l'onglet DNS
  2. Ajouter un enregistrement A :
    • Type : A
    • Nom : n8n (ou @ pour le domaine racine)
    • Adresse IPv4 : Votre IP publique ou utilisez le proxy Cloudflare
    • Statut du proxy : ✓ Proxifié (nuage orange)

Ajouter l'Hôte Proxy dans NPM

  1. Accédez à Nginx Proxy Manager : http://<npm-ip>:81
  2. Cliquez sur « Hôtes Proxy » → « Ajouter un Hôte Proxy »
  3. Onglet Détails :
    • Noms de domaine : n8n.my-portfolio.tk
    • Schéma : http
    • Nom d'hôte/IP de transfert : n8n (nom du conteneur Docker) ou IP
    • Port de transfert : 5678
    • Support Websockets : ✓ CRITIQUE pour n8n
    • Bloquer les Exploits Courants : ✓
  4. Onglet SSL :

    • Sélectionnez « Demander un nouveau Certificat SSL »

    • Activez « Utiliser un Challenge DNS »

    • Fournisseur DNS : Sélectionnez Cloudflare

    • Contenu du Fichier d'Identification :

      dns_cloudflare_api_token = YOUR_CLOUDFLARE_API_TOKEN

    • Adresse E-mail : Votre e-mail pour Let’s Encrypt

    • Options :

      • ✓ Forcer SSL (Force SSL)
      • ✓ Support HTTP/2 (HTTP/2 Support)
      • ✓ HSTS Activé (HSTS Enabled)
  5. Cliquez sur Sauvegarder

Étape 4 : Vérifier l'Accès

# Test DNS resolution
nslookup n8n.my-portfolio.tk

# Test HTTPS access
curl -I https://n8n.my-portfolio.tk

# Access in browser
https://n8n.my-portfolio.tk

Avantages et Limites

✅ Avantages : - Certificat SSL public valide - Renouvellement automatique tous les 90 jours - Pas besoin d'ouvrir les ports 80/443 pour la validation HTTP-01 - Fonctionne de n'importe où avec un accès Internet

⚠️ Limites : - Nécessite un domaine public et la propagation DNS - Dépend de services externes (Cloudflare, Let’s Encrypt) - Problèmes potentiels de confidentialité liés à l'exposition publique

Option B : DNS Local (pfSense) + Certificat Local

Cas d'Utilisation

Idéal pour : - Accès homelab interne uniquement - Aucun domaine public souhaité - Environnements hors ligne ou isolés (air-gapped) - Développement et tests

Étape 1 : Configuration du Split-DNS dans pfSense

Configurer les Surcharges d'Hôte du Résolveur DNS

  1. Connectez-vous à l'interface web pfSense
  2. Naviguez vers Services → DNS Resolver
  3. Assurez-vous que le Résolveur DNS est activé
  4. Faites défiler jusqu'à Host Overrides
  5. Cliquez sur « Ajouter » :
    • Hôte : n8n
    • Domaine : local.home (ou votre choix)
    • Adresse IP : IP de l'hôte exécutant NPM
    • Description : plateforme d'automatisation n8n
  6. Cliquez sur Sauvegarder → Appliquer les Changements

Tester la Résolution DNS

À partir d'un client sur votre réseau :

# Should resolve to your NPM host IP
nslookup n8n.local.home

# Or using dig
dig n8n.local.home

Étape 2 : Générer un Certificat SSL Local

Option A : Certificat Auto-Signé (OpenSSL)

# Generate private key
openssl genrsa -out n8n-local.key 2048

# Generate certificate signing request
openssl req -new -key n8n-local.key -out n8n-local.csr \
  -subj "/C=US/ST=State/L=City/O=Homelab/CN=n8n.local.home"

# Generate self-signed certificate
openssl x509 -req -days 365 -in n8n-local.csr \
  -signkey n8n-local.key -out n8n-local.crt

Option B : mkcert (Recommandé)

Pourquoi mkcert ? Fait automatiquement confiance par les navigateurs après l'installation de l'AC.

# Install mkcert
# On Ubuntu/Debian:
sudo apt install libnss3-tools
wget https://github.com/FiloSottile/mkcert/releases/download/v1.4.4/mkcert-v1.4.4-linux-amd64
sudo mv mkcert-v1.4.4-linux-amd64 /usr/local/bin/mkcert
sudo chmod +x /usr/local/bin/mkcert

# Install local CA
mkcert -install

# Generate certificate for n8n
mkcert n8n.local.home "*.local.home"

# This creates:
# n8n.local.home+1.pem (certificate)
# n8n.local.home+1-key.pem (private key)

Étape 3 : Importer le Certificat dans NPM

  1. Dans Nginx Proxy Manager, allez à Certificats SSL
  2. Cliquez sur « Ajouter un Certificat SSL » → « Personnalisé »
  3. Téléchargez les fichiers :
    • Clé du Certificat : n8n.local.home+1-key.pem
    • Certificat : n8n.local.home+1.pem
  4. Cliquez sur Sauvegarder

Étape 4 : Créer l'Hôte Proxy

  1. Cliquez sur « Hôtes Proxy » → « Ajouter un Hôte Proxy »
  2. Onglet Détails :
    • Noms de domaine : n8n.local.home
    • Schéma : http
    • Nom d'hôte/IP de transfert : n8n ou IP du conteneur
    • Port de transfert : 5678
    • Support Websockets : ✓
    • Bloquer les Exploits Courants : ✓
  3. Onglet SSL :
    • Certificat SSL : Sélectionnez votre certificat personnalisé
    • Forcer SSL : ✓
    • Support HTTP/2 : ✓
  4. Cliquez sur Sauvegarder

Étape 5 : Installer l'AC sur les Appareils Clients

Pour mkcert :

Linux :

mkcert -install

Windows :

# Run mkcert installer
mkcert-v1.4.4-windows-amd64.exe -install

macOS :

mkcert -install

Appareils Mobiles : - Exportez le certificat d'AC depuis mkcert -CAROOT - Transférez vers l'appareil - Installez comme certificat de confiance

Avantages et Limites

✅ Avantages : - Aucune dépendance externe - Fonctionne hors ligne - Contrôle total sur les certificats - Aucune exposition publique

⚠️ Limites : - Nécessite l'installation de l'AC sur tous les appareils clients - Renouvellement manuel du certificat - Uniquement accessible sur le réseau local

Dépannage des Problèmes Courants

Problème 1 : Erreur « Secure Cookie » de n8n

Problème : n8n nécessite HTTPS pour les cookies sécurisés

Solutions :

# Option 1: Use HTTPS (recommended)
# Follow steps above

# Option 2: Disable secure cookie (TESTING ONLY)
docker run -e N8N_SECURE_COOKIE=false n8nio/n8n

Problème 2 : NPM ne peut pas Résoudre le Nom du Conteneur

Problème : Nginx Proxy Manager ne parvient pas à atteindre le conteneur n8n

Solutions :

# Option 1: Use Docker network
# Ensure both containers are on same Docker network

# Option 2: Use host IP instead of container name
# In NPM proxy host, use: 192.168.x.x instead of "n8n"

# Option 3: Use Docker bridge IP
docker inspect n8n | grep IPAddress
# Use that IP in NPM configuration

Problème 3 : Le DNS Interne ne Résout pas

Problème : Les clients ne peuvent pas résoudre n8n.local.home

Solutions :

  1. Vérifier que le Résolveur DNS pfSense est en cours d'exécution :

    • Services → DNS Resolver → Status

  2. Vérifier les paramètres DNS du client :

    # Linux/macOS
cat /etc/resolv.conf
# Should show pfSense IP

# Windows
ipconfig /all
# Check DNS Servers

  3. Tester à partir de pfSense lui-même :

    # Diagnostics → Command Prompt
nslookup n8n.local.home

Problème 4 : Échec du Challenge DNS-01 de Let’s Encrypt

Problème : La demande de certificat échoue

Solutions :

  1. Vérifier les Permissions du Jeton API :

    • Doit avoir Zone:DNS:Edit pour la zone spécifique
    • Ne peut pas utiliser la clé API globale

  2. Vérifier la Propagation de l'Enregistrement TXT :

    dig _acme-challenge.n8n.my-portfolio.tk TXT

  3. Examiner les Logs NPM :

    docker logs nginx-proxy-manager

Meilleures Pratiques de Sécurité

1. Activer l'Authentification dans n8n

# docker-compose.yml
environment:
  - N8N_BASIC_AUTH_ACTIVE=true
  - N8N_BASIC_AUTH_USER=admin
  - N8N_BASIC_AUTH_PASSWORD=secure_password_here

2. Restreindre l'Accès par IP

Dans NPM, ajoutez une Liste d'Accès : - Autoriser : Vos IP de confiance - Refuser : Toutes les autres

3. Utiliser des Paramètres Cloudflare Robustes

  • Activer le Mode de Lutte contre les Bots (Bot Fight Mode)
  • Ajouter des Règles de Page (Page Rules) pour la limitation du débit
  • Activer Toujours Utiliser HTTPS (Always Use HTTPS)

4. Surveiller les Logs d'Accès

# View NPM access logs
docker exec -it nginx-proxy-manager tail -f /data/logs/proxy-host-*.log

5. Surveillance Régulière des Certificats

  • Définir un rappel dans le calendrier pour l'expiration du certificat
  • Surveiller les logs de renouvellement NPM
  • Tester l'accès après les renouvellements

Conclusion

Sécuriser n8n avec une configuration DNS et TLS appropriée transforme une plateforme d'automatisation vulnérable en un système prêt pour la production. Que vous choisissiez l'approche du domaine public avec Let’s Encrypt ou la méthode du certificat local, les deux offrent un chiffrement solide et une gestion des accès professionnelle.

La clé est de comprendre vos besoins : - Accès public nécessaire ? → Option A (Cloudflare + Let’s Encrypt) - Usage interne uniquement ? → Option B (pfSense + Certificats Locaux)

Les deux approches démontrent une maturité de l'infrastructure et une sensibilisation à la sécurité, des compétences essentielles pour tout administrateur de homelab ou professionnel de la sécurité.

Rappel : La sécurité est une question de couches. Le DNS et le TLS ne sont que le début - ajoutez l'authentification, la surveillance et des audits réguliers pour une protection complète.
