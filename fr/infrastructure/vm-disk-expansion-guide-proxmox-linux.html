Guide d'expansion de disque de VM (Proxmox + Linux)

Tags: #proxmox #storage #filesystem #resize

Référence Rapide

L'expansion d'un disque de VM est un processus en 3 étapes: 1. Proxmox : Augmenter la taille du disque virtuel 2. OS Invité : Étendre la partition 3. OS Invité : Redimensionner le système de fichiers

Étape 1 : Augmenter la taille du disque dans Proxmox

Via l'interface web

1. Sélectionnez VM → Hardware
2. Cliquez sur Disque Dur (scsi0)
3. Cliquez sur le bouton "Resize disk"
4. Entrez la taille à ajouter (ex. : +50)
5. Cliquez sur "Resize disk"

Via la CLI (Hôte Proxmox)

# Increase VM 100's disk by 50GB
qm resize 100 scsi0 +50G

# Verify
qm config 100 | grep scsi0

Résultat : Le disque virtuel est plus grand, mais l'OS invité ne le voit pas encore.

Étape 2 : Étendre la partition (À l'intérieur de la VM)

Vérifier la situation actuelle

# View disk and partition layout
lsblk

# Exemple de sortie :
# sda 8:0 0 150G 0 disk ← Le disque est de 150 Go
# └─sda2 8:2 0 100G 0 part / ← La partition est toujours de 100 Go

Étendre la partition

Méthode A : Utilisation de growpart (Recommandée)

# Installer si nécessaire
sudo apt install cloud-guest-utils

# Étendre la partition 2 sur /dev/sda
sudo growpart /dev/sda 2

# Vérifier
lsblk
# sda2 devrait maintenant afficher 150 Go

Méthode B : Utilisation de parted

# Mode interactif
sudo parted /dev/sda
(parted) print free # Afficher la disposition
(parted) resizepart 2 100% # Redimensionner la partition 2 à 100 %
(parted) quit

# Mode non-interactif
sudo parted /dev/sda resizepart 2 100%

Méthode C : Utilisation de fdisk (Avancé)

# Uniquement si les méthodes ci-dessus échouent
sudo fdisk /dev/sda
# Type: d, 2 (supprimer la partition 2)
# Type: n, p, 2, <enter>, <enter> (recréer plus grande)
# Type: w (écrire les changements)
# Redémarrer si la partition est utilisée

Étape 3 : Redimensionner le système de fichiers (À l'intérieur de la VM)

Pour le système de fichiers ext4 (Ubuntu/Debian)

# Redimensionner le système de fichiers pour remplir la partition
sudo resize2fs /dev/sda2

# Vérifier
df -h /

Pour le système de fichiers XFS (CentOS/RHEL)

# Redimensionner le système de fichiers XFS
sudo xfs_growfs /

# Vérifier
df -h /

Pour le système de fichiers Btrfs

# Redimensionner le système de fichiers Btrfs
sudo btrfs filesystem resize max /

# Vérifier
df -h /

Exemple de procédure complète

Scénario : Étendre une VM Ubuntu de 100 Go à 150 Go

Étape 1 : Hôte Proxmox

# Ajouter 50 Go à la VM 100
qm resize 100 scsi0 +50G

Étape 2 : À l'intérieur de la VM Ubuntu

# Vérifier l'état actuel
lsblk
df -h /

# La sortie affiche :
# Disque : 150 Go, Partition : 100 Go, Système de fichiers : 99 Go

# Étendre la partition
sudo growpart /dev/sda 2

# Vérifier l'extension de la partition
lsblk
# La sortie affiche maintenant : Partition : 150 Go

# Étendre le système de fichiers
sudo resize2fs /dev/sda2

# Vérifier le résultat final
df -h /
# La sortie affiche maintenant : Système de fichiers : 148 Go (150 Go - surdébit)

Succès ! Votre VM dispose maintenant de 50 Go d'espace supplémentaire.

Dépannage

Problème : La partition ne s'étend pas

Symptôme :

sudo growpart /dev/sda 2
# Error: no space to grow

Solution :

# Vérifiez si le redimensionnement Proxmox a fonctionné
lsblk
# Regardez la taille de sda - elle devrait être plus grande

# Si la taille de sda est toujours l'ancienne :
# - Arrêtez la VM
# - Redimensionnez à nouveau dans Proxmox
# - Démarrez la VM

Problème : Avertissement "GPT PMBR size mismatch"

Symptôme :

sudo fdisk -l /dev/sda
# GPT PMBR size mismatch (209715199 != 314572799)

Solution : Ceci est normal et sera corrigé automatiquement lorsque vous redimensionnerez la partition.

# Corriger la table GPT
sudo parted /dev/sda
(parted) print
# Répondez 'Fix' lorsque vous y êtes invité
(parted) quit

Problème : "Le système de fichiers est monté, impossible de redimensionner"

Pour ext4 :

# ext4 prend en charge le redimensionnement en ligne
sudo resize2fs /dev/sda2
# Ceci fonctionne même monté

Pour XFS :

# XFS doit être monté pour redimensionner
sudo xfs_growfs /
# Utilise le point de montage, pas le périphérique

Pour les autres :

# Peut nécessiter de démonter ou d'utiliser un CD live
# Démarrer depuis une clé USB live
# Exécuter resize2fs /dev/sda2
# Redémarrer

Problème : La partition n'utilise pas l'intégralité du disque

Vérifiez l'espace non alloué :

sudo parted /dev/sda print free

# Affiche quelque chose comme :
# Number Start End Size File system Name
# 1 1049kB 2097kB 1049kB 
# 2 2097kB 107GB 107GB ext4
# 107GB 161GB 53.7GB Espace Libre ← Non alloué

Solution : Étendre la partition (Étape 2 ci-dessus)

Différentes conventions de nommage des disques

Type de disque VM Périphérique Linux Utilisation
IDE /dev/hda Ancien, lent
SCSI /dev/sda Courant
VirtIO Block /dev/vda Le plus rapide (recommandé)
VirtIO SCSI /dev/sda Rapide + support TRIM

Ajustez les commandes en conséquence : - VirtIO : Utilisez /dev/vda au lieu de /dev/sda - Disques multiples : /dev/sdb, /dev/sdc, etc.

Liste de vérification préliminaire

Avant d'étendre le disque :

  • # Dans Proxmox
qm config <vmid> | grep scsi0

# Dans la VM
df -h /
lsblk

Vérification Post-Expansion

# Dans la VM - tout vérifier
lsblk # Tailles du disque et de la partition
df -h # Taille et utilisation du système de fichiers
sudo parted /dev/sda print # Table de partition

# Tout devrait afficher les nouvelles tailles plus grandes

Tailles de disque courantes

Original Ajouter Final Cas d'utilisation
32G +18G 50G Serveur minimal
100G +50G 150G VM de développement
100G +100G 200G Charge de travail de production
200G +300G 500G Base de données/stockage

Conseil : Ajoutez de l'espace par incréments. Il est plus facile d'ajouter plus tard que de retirer.

Scénario LVM (Processus différent)

Si votre VM utilise LVM :

# Vérifier si LVM
lsblk
# Cherchez "lvm" dans la colonne TYPE

# Exemple avec LVM
lsblk
# sda
# └─sda2 (partition physique)
# └─ubuntu--vg-root (volume LVM)

# Processus de redimensionnement :
# 1. Proxmox : Augmenter la taille du disque
qm resize 100 scsi0 +50G

# 2. VM : Étendre la partition
sudo growpart /dev/sda 2

# 3. VM : Redimensionner le volume physique
sudo pvresize /dev/sda2

# 4. VM : Étendre le volume logique
sudo lvextend -l +100%FREE /dev/ubuntu-vg/root

# 5. VM : Redimensionner le système de fichiers
sudo resize2fs /dev/ubuntu-vg/root

# Vérifier
df -h /

Référence rapide des commandes

# Visualiser la disposition du disque
lsblk
lsblk -f # Avec les informations du système de fichiers
sudo parted /dev/sda print # Informations détaillées sur la partition
sudo fdisk -l /dev/sda # Vue alternative

# Vérifier le système de fichiers
df -h # Systèmes de fichiers montés
df -h / # Spécifiquement le système de fichiers racine
sudo tune2fs -l /dev/sda2 # Détails ext4

# Opérations d'expansion
sudo growpart /dev/sda 2 # Étendre la partition
sudo resize2fs /dev/sda2 # Étendre ext4
sudo xfs_growfs / # Étendre XFS
sudo btrfs filesystem resize max / # Étendre Btrfs

# Vérifier
lsblk && df -h / # Vérification rapide des deux

Exemple Concret

État Initial :

# Proxmox : Disque VM = 100 Go
# VM : Partition = 100 Go
# VM : Système de fichiers = 99 Go (22 Go utilisés)

Commandes utilisées :

# 1. Proxmox (via l'interface web) :
# Hardware → Disk → Resize → +50

# 2. VM Ubuntu :
sudo apt install cloud-guest-utils
sudo growpart /dev/sda 2
sudo resize2fs /dev/sda2

État Final :

# Proxmox : Disque VM = 150 Go
# VM : Partition = 150 Go
# VM : Système de fichiers = 148 Go (22 Go utilisés, 120 Go libres)

Résultat : +49 Go d'espace utilisable gagné ! ✅
