Routine de maintenance des VM Proxmox

Tags: #proxmox #virtualization #maintenance #automation

Vérification de l'état du stockage

Surveillance du stockage de l'hôte

# Check root filesystem usage (keep below 90%)
df -h /

# Target: Keep usage below 85-90%
# Alert level: 90%+
# Critical: 95%+

Surveillance du Thin Pool LVM

# Check thin pool data usage
lvs pve/data

# Target: Keep data% below 80%
# Alert level: 80%+
# Critical: 90%+

# Full view with all volumes
lvs -a | grep data

Extension du stockage (si nécessaire)

# Check available space in volume group
vgdisplay pve | grep -i free

# Extend root filesystem
lvextend -l +100%FREE /dev/pve/root
resize2fs /dev/pve/root

# Verify expansion
df -h /

Stratégie de gestion des Snapshots

⚠️ Règle d'Or

Maintenez un maximum de 1 à 2 snapshots par VM à tout moment

Types de Snapshots

1. Snapshot de Référence (Golden Snapshot - À Conserver)

  • Créé après une configuration majeure réussie
  • Représente un état stable et fonctionnel
  • Mettre à jour rarement (trimestriellement ou après des changements majeurs)
# Create golden snapshot
qm snapshot <VMID> golden-working-state

# Update golden (delete old, create new)
qm delsnapshot <VMID> golden-working-state
qm snapshot <VMID> golden-working-state

2. Snapshot Pré-Changement (Temporaire)

  • Créé avant les opérations risquées
  • Supprimez dans les 24 à 48 heures
  • Ne jamais en conserver plus d'un
# Avant un changement risqué
qm snapshot <VMID> pre-change-$(date +%Y%m%d)

# Après un changement réussi (SUPPRIMEZ-LE !)
qm delsnapshot <VMID> pre-change-20251030

# Si le changement a échoué (rollback puis suppression)
qm rollback <VMID> pre-change-20251030
qm delsnapshot <VMID> pre-change-20251030

Lister et Nettoyer les Snapshots

# List all snapshots for a VM
qm listsnapshot <VMID>

# Clean up old snapshots
qm delsnapshot <VMID> <snapshot-name>

# Example cleanup script
for snap in $(qm listsnapshot 100 | grep -v current | grep -v golden | awk '{print $2}'); do
    qm delsnapshot 100 $snap
done

Optimisation des performances des VM

Configuration du disque

# Paramètres de disque optimaux pour la VM 100
qm set 100 -scsi0 local-lvm:vm-100-disk-0,cache=writeback,iothread=1

# Modes de cache (par ordre de vitesse) :
# - writeback (le plus rapide, recommandé)
# - writethrough (équilibré)
# - directsync (le plus lent, à éviter sauf nécessité)

Vérification des performances du disque de la VM (à l'intérieur de la VM)

# À l'intérieur de la VM, vérifier l'I/O du disque
iostat -x 1 10

# Surveiller :
# - %util > 80% = goulot d'étranglement du disque
# - await > 50ms = réponses lentes
# - iowait% élevé = problèmes d'I/O

Vérification de la mémoire et du Swap (à l'intérieur de la VM)

# Vérifier l'utilisation de la mémoire
free -h

# Surveiller l'utilisation du swap (devrait être 0)
vmstat 1

# Regarder les colonnes 'si' et 'so' - elles devraient être à 0

Protection du Thin Pool LVM

Activer l'extension automatique

Empêche le thin pool de se remplir complètement et de faire planter les VM.

# Sauvegarder la configuration
cp /etc/lvm/lvm.conf /etc/lvm/lvm.conf.backup

# Activer l'extension automatique
sed -i 's/# thin_pool_autoextend_threshold = 100/thin_pool_autoextend_threshold = 80/' /etc/lvm/lvm.conf
sed -i 's/# thin_pool_autoextend_percent = 20/thin_pool_autoextend_percent = 20/' /etc/lvm/lvm.conf

# Vérifier les paramètres
grep thin_pool_autoextend /etc/lvm/lvm.conf

Sortie attendue :

thin_pool_autoextend_threshold = 80
thin_pool_autoextend_percent = 20

Stratégie de sauvegarde

Pourquoi les sauvegardes sont supérieures aux Snapshots

  • Les Snapshots ne sont PAS des sauvegardes
  • Les Snapshots résident sur le même stockage (point de défaillance unique)
  • Les sauvegardes peuvent être restaurées n'importe où
  • Les sauvegardes survivent aux pannes matérielles

Mise en place des sauvegardes automatisées

Via l'interface web :

  1. Datacenter → Sauvegarde (Backup) → Ajouter (Add)
  2. Configurer:
    • Planification : Quotidien à 2h00
    • Mode de sélection : Inclure les VM sélectionnées
    • Conservation : Quotidienne : 3, Hebdomadaire : 2, Mensuelle : 1
    • Compression : ZSTD
    • Mode : Snapshot

Via CLI :

# Sauvegarde manuelle
vzdump <VMID> --mode snapshot --compress zstd --storage local

# Sauvegarde vers un stockage externe (recommandé)
vzdump <VMID> --storage backup-storage --compress zstd

Stratégie de rétention des sauvegardes

Quotidienne : Conserver 3 (aujourd'hui, hier, il y a 2 jours)
Hebdomadaire : Conserver 2 (les 2 dernières semaines)
Mensuelle : Conserver 1 (le mois dernier)

Dépannage d'urgence

Liste de contrôle en cas de blocage de VM

1. Vérifier le stockage de l'hôte (le plus fréquent)

df -h /
# Si > 90 %, étendre ou nettoyer immédiatement

2. Vérifier le Thin Pool

lvs pve/data
# Si data% > 80 %, supprimer les anciens snapshots

3. Vérifier le cache du disque de la VM

qm config <VMID> | grep cache
# Devrait être : cache=writeback

4. À l'intérieur de la VM : Vérifier l'I/O

iostat -x 1
# Rechercher des %util ou des temps d'attente élevés

5. À l'intérieur de la VM : Vérifier la mémoire

free -h
vmstat 1
# Vérifier l'utilisation du swap (colonnes si/so)

Commandes de nettoyage rapide

# Nettoyage de l'hôte
apt clean
apt autoclean
journalctl --vacuum-time=7d
rm -rf /var/cache/apt/archives/*

# Trouver les fichiers volumineux
du -h / --max-depth=3 2>/dev/null | sort -h | tail -20

# Supprimer les anciens noyaux (garder le noyau actuel + 1)
dpkg --list | grep linux-image
apt remove linux-image-X.X.X-X-pve

Scripts d'automatisation

Script de vérification hebdomadaire de l'état

#!/bin/bash
# Enregistrer sous : /root/proxmox-health-check.sh

echo "=== Proxmox Health Check - $(date) ===" | tee -a /var/log/proxmox-health.log

# Vérifier le stockage racine
echo "Root Storage:" | tee -a /var/log/proxmox-health.log
df -h / | grep mapper | tee -a /var/log/proxmox-health.log

# Vérifier le thin pool
echo "Thin Pool Usage:" | tee -a /var/log/proxmox-health.log
lvs pve/data | tee -a /var/log/proxmox-health.log

# Lister les snapshots pour chaque VM
for vmid in $(qm list | awk 'NR>1 {print $1}'); do
    echo "VM $vmid snapshots:" | tee -a /var/log/proxmox-health.log
    qm listsnapshot $vmid | tee -a /var/log/proxmox-health.log
done

# Alerte si le stockage > 85%
USAGE=$(df / | awk 'NR==2 {print $5}' | sed 's/%//')
if [ $USAGE -gt 85 ]; then
    echo "⚠️  AVERTISSEMENT : Stockage racine à ${USAGE}%" | tee -a /var/log/proxmox-health.log
fi

echo "===================================" | tee -a /var/log/proxmox-health.log

Rendre exécutable et planifier

chmod +x /root/proxmox-health-check.sh

# Ajouter à crontab (chaque dimanche à 8h00)
crontab -e
# Ajouter cette ligne :
0 8 * * 0 /root/proxmox-health-check.sh

Seuils de surveillance

Définir les alertes lorsque :

Métriques Avertissement Critique Action
Stockage Racine 85% 95% Étendre ou nettoyer
Thin Pool Data 75% 85% Supprimer les snapshots
Thin Pool Meta 75% 85% Alerter l'administrateur
Attente I/O Disque VM 20% 40% Vérifier le mode de cache
Nombre de Snapshots 3 par VM 5 par VM Nettoyer immédiatement

Leçons Apprises

Cause profonde des blocages précédents

  1. ✅ Système de fichiers racine plein à 100 % - cause principale
  2. ✅ Mode de cache : directsync - I/O lent
  3. ✅ Chaîne de Snapshots - inefficacité de l'espace
  4. ✅ Absence d'extension automatique du thin pool - risque de saturation

Solutions Appliquées

  1. ✅ Extension de la racine de 96 Go → 112 Go
  2. ✅ Changement du cache : directsync → writeback
  3. ✅ Suppression de la chaîne de snapshots inutiles
  4. ✅ Activation de l'extension automatique du thin pool
  5. ✅ Établissement d'une stratégie de gestion des snapshots

Stratégie de Prévention

  • Surveiller le stockage hebdomadairement
  • Ne jamais dépasser 2 snapshots par VM
  • Utiliser des sauvegardes, pas des snapshots, pour la sécurité
  • Activer les alertes à 85 % d'utilisation du stockage

Référence rapide des commandes

# Stockage
df -h /                                    # Vérifier le stockage racine

free -h                                   # Vérifier la mémoire
vmstat 1                                  # Statistiques système

apt clean                                 # Nettoyer le cache des paquets
