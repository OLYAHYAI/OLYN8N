Difficultés lors de l'extension du disque d'une VM Ubuntu sur Proxmox

2025-11-19
5 min de lecture
Infrastructure
homelab

Difficultés : Extension du disque de ma VM Ubuntu sur Proxmox

Le Problème

Vous avez augmenté la taille du disque dans Proxmox (de 32 Go à 100 Go), mais Ubuntu n'affiche toujours que 32 Go disponibles lorsque vous exécutez df -h. Le disque a été redimensionné au niveau de l'hyperviseur, mais le système d'exploitation n'a pas encore reconnu le nouvel espace.

Il s'agit d'un scénario courant dans la gestion d'un homelab, et comprendre la solution appropriée démontre une compétence en matière d'infrastructure.

Comprendre les Couches

Lors du redimensionnement des disques de VM, vous travaillez avec trois couches distinctes :

Layer 1: Hypervisor (Proxmox)
   ↓ Le redimensionnement du disque se produit ici (32 Go → 100 Go)
Layer 2: Table de partition
   ↓ La partition doit être étendue
Layer 3: Système de fichiers
   ↓ Le système de fichiers doit être étendu pour remplir la partition

Chaque couche doit être mise à jour séparément.

Situation Initiale

Avant le Redimensionnement :

  • Proxmox affiche : Disque de 32 Go
  • Ubuntu affiche : 32 Go disponibles sur /

Après le Redimensionnement Proxmox :

  • Proxmox affiche : Disque de 100 Go ✓
  • Ubuntu affiche : 32 Go disponibles sur / ❌

Écart: 68 Go d'espace non alloué

Résolution Étape par Étape

Étape 1 : Vérifier la Nouvelle Taille du Disque

# Check disk and partition layout
lsblk

# Example output:
NAME   MAJ:MIN RM  SIZE RO TYPE MOUNTPOINT
sda      8:0    0  100G  0 disk           # ✓ Disk shows 100G
├─sda1   8:1    0    1M  0 part           # EFI/BIOS boot
└─sda2   8:2    0   32G  0 part /         # ❌ Partition still 32G

Observation Clé: - sda (disque) affiche 100 Go ✓ - sda2 (partition) affiche 32 Go ❌ - Espace non alloué : 68 Go (100 Go - 32 Go)

Étape 2 : Installer les Outils Nécessaires

# Install cloud-guest-utils (provides growpart)
sudo apt update
sudo apt install cloud-guest-utils -y

# Alternative: Install parted (if growpart unavailable)
sudo apt install parted -y

Étape 3 : Étendre la Partition

Utilisation de growpart (Recommandé) :

# Extend partition 2 on /dev/sda to fill available space
sudo growpart /dev/sda 2

# Expected output:
# CHANGED: partition=2 start=2048 old: size=67106816 end=67108864 new: size=209713119 end=209715167

Utilisation de parted (Alternative) :

# Launch parted
sudo parted /dev/sda

# Inside parted:
(parted) print                    # Show current layout
(parted) resizepart 2 100%        # Resize partition 2 to 100% of disk
(parted) quit                     # Exit parted

# Verify
lsblk

Étape 4 : Étendre le Système de Fichiers

La partition est maintenant plus grande, mais le système de fichiers pense toujours qu'il ne fait que 32 Go. Nous devons étendre le système de fichiers pour remplir la partition.

Pour les Systèmes de Fichiers ext4 (Le plus courant) :

# Resize ext4 filesystem on partition 2
sudo resize2fs /dev/sda2

# Expected output:
# resize2fs 1.46.5 (30-Dec-2021)
# Filesystem at /dev/sda2 is mounted on /; on-line resizing required
# old_desc_blocks = 4, new_desc_blocks = 13
# The filesystem on /dev/sda2 is now 26214139 (4k) blocks long.

Pour les Systèmes de Fichiers XFS :

# Resize XFS filesystem mounted on /
sudo xfs_growfs /

# Or specify the device
sudo xfs_growfs /dev/sda2

Pour les Volumes Logiques LVM :

Si vous utilisez LVM (vérifiez avec lsblk) :

# Extend physical volume
sudo pvresize /dev/sda2

# Extend logical volume (replace vg_name/lv_name with your actual names)
sudo lvextend -l +100%FREE /dev/vg_name/lv_name

# Resize filesystem
sudo resize2fs /dev/vg_name/lv_name  # for ext4
# or
sudo xfs_growfs /  # for XFS

Étape 5 : Vérifier les Changements

# Check disk usage
df -h

# Expected output:
# Filesystem      Size  Used Avail Use% Mounted on
# /dev/sda2        98G   15G   79G  16% /          # ✓ Now shows ~100G!

# Verify with lsblk
lsblk

# Should show:
# sda      8:0    0  100G  0 disk
# ├─sda1   8:1    0    1M  0 part
# └─sda2   8:2    0  100G  0 part /         # ✓ Partition now 100G

Comprendre les Commandes

growpart

  • Objectif: Étend la partition pour remplir l'espace disque disponible
  • Sûr: Non destructif, préserve les données existantes
  • Utilisation: growpart <device> <partition-number>
  • Example: growpart /dev/sda 2

resize2fs

  • Objectif: Étend les systèmes de fichiers ext2/ext3/ext4
  • En ligne: Fonctionne sur les systèmes de fichiers montés
  • Utilisation: resize2fs <partition-device>
  • Example: resize2fs /dev/sda2

xfs_growfs

  • Objectif: Étend les systèmes de fichiers XFS
  • Point de Montage: Nécessite le point de montage du système de fichiers (pas le périphérique)
  • Utilisation: xfs_growfs <mount-point>
  • Example: xfs_growfs /

Scénarios Courants et Solutions

Scénario 1 : Table de Partition UEFI/GPT

# Check partition table type
sudo parted /dev/sda print

# If GPT, growpart works automatically
sudo growpart /dev/sda 2
sudo resize2fs /dev/sda2

Scénario 2 : Table de Partition MBR/MSDOS

# Verify partition table
sudo fdisk -l /dev/sda

# Use parted for MBR tables
sudo parted /dev/sda resizepart 2 100%
sudo resize2fs /dev/sda2

Scénario 3 : Partitions Multiples

# Identify which partition to extend
lsblk -f

# Extend the correct partition (usually root /)
sudo growpart /dev/sda <partition-number>
sudo resize2fs /dev/sda<partition-number>

Scénario 4 : LVM avec Thin Provisioning

# Extend physical volume
sudo pvresize /dev/sda2

# Check volume groups
sudo vgdisplay

# Extend logical volume
sudo lvextend -l +100%FREE /dev/ubuntu-vg/ubuntu-lv

# Resize filesystem
sudo resize2fs /dev/ubuntu-vg/ubuntu-lv  # for ext4
# or
sudo xfs_growfs /  # for XFS

Dépannage

Erreur : « growpart: command not found »

# Install cloud-guest-utils
sudo apt update
sudo apt install cloud-guest-utils -y

Erreur : « resize2fs: Bad magic number »

Cause: La partition n'est pas ext2/ext3/ext4

Solution: Vérifiez le type de système de fichiers et utilisez l'outil approprié

# Check filesystem type
lsblk -f

# For XFS:
sudo xfs_growfs /

# For btrfs:
sudo btrfs filesystem resize max /

Erreur : « The filesystem is already X blocks long »

Cause: Le système de fichiers est déjà étendu (le redimensionnement de la partition a peut-être échoué)

Solution: Vérifiez que la partition a bien été étendue

# Check partition size
sudo parted /dev/sda print

# If partition is still small, retry growpart
sudo growpart /dev/sda 2

Avertissement : « Partition X is being used »

Cause: La partition est montée (normal pour la partition root)

Solution: Ceci est attendu et sans danger. Les opérations de redimensionnement fonctionnent en ligne pour ext4 et XFS.

# Continue with resize - it's safe
sudo resize2fs /dev/sda2

Automatisation avec des Scripts

Script de Redimensionnement Automatisé

#!/bin/bash
# resize-vm-disk.sh
# Automatically extends partition and filesystem after Proxmox resize

set -e  # Exit on error

DISK="/dev/sda"
PARTITION="2"

echo "Starting disk resize process..."

# Install required tools if needed
if ! command -v growpart &> /dev/null; then
    echo "Installing cloud-guest-utils..."
    apt update && apt install -y cloud-guest-utils
fi

# Show current state
echo "Current disk layout:"
lsblk $DISK

# Extend partition
echo "Extending partition ${PARTITION}..."
growpart $DISK $PARTITION

# Detect filesystem type
FS_TYPE=$(lsblk -no FSTYPE ${DISK}${PARTITION})

# Extend filesystem based on type
echo "Extending $FS_TYPE filesystem..."
case $FS_TYPE in
    ext4|ext3|ext2)
        resize2fs ${DISK}${PARTITION}
        ;;
    xfs)
        xfs_growfs /
        ;;
    *)
        echo "Unsupported filesystem: $FS_TYPE"
        exit 1
        ;;
esac

# Show final state
echo "Final disk layout:"
lsblk $DISK
df -h /

Utilisation :

chmod +x resize-vm-disk.sh
sudo ./resize-vm-disk.sh

Meilleures Pratiques

1. Planifier la Croissance

Lors de la création de VMs : - Commencez avec une taille raisonnable - Utilisez le thin provisioning lorsque cela est possible - Documentez les tailles de disque dans votre inventaire homelab

2. Sauvegarder Avant le Redimensionnement

# Take Proxmox snapshot before resize
# In Proxmox UI: VM → Snapshots → Take Snapshot

# Or use command line
qm snapshot <vmid> pre-resize --description "Before disk resize"

3. Vérifier l'Espace Libre sur l'Hôte

# Before resizing, check Proxmox host has space
df -h

# Check storage pool
pvesm status

4. Utiliser les Images Cloud-Init

Les images Cloud incluent souvent l'expansion automatique du disque :

# Example cloud-init config
growpart:
  mode: auto
  devices: ['/']
  ignore_growroot_disabled: false

Comprendre les Types de Disques dans Proxmox

SCSI vs VirtIO vs SATA

Différents contrôleurs de disque affectent les noms des périphériques :

  • SCSI: /dev/sda, /dev/sdb
  • VirtIO: /dev/vda, /dev/vdb
  • SATA: /dev/sda, /dev/sdb

Ajustez les commandes en conséquence :

# For VirtIO disks:
sudo growpart /dev/vda 1
sudo resize2fs /dev/vda1

# For SCSI/SATA disks:
sudo growpart /dev/sda 2
sudo resize2fs /dev/sda2

Conclusion

L'extension des disques de VM après un redimensionnement Proxmox est un processus en trois étapes :

  1. Niveau Proxmox: Redimensionner le disque virtuel (fait via l'interface utilisateur)
  2. Niveau Partition: Étendre la partition avec growpart
  3. Niveau Système de Fichiers: Étendre le système de fichiers avec resize2fs ou xfs_growfs

Ce processus est : - ✅ Non destructif : Aucune perte de données - ✅ En ligne : Fonctionne sur les systèmes de fichiers montés - ✅ Réversible : Les snapshots offrent un filet de sécurité - ✅ Automatisable : Peut être scripté pour la cohérence

Comprendre ce processus est essentiel pour la gestion d'un homelab et démontre des compétences adéquates en administration d'infrastructure.

Conseil de pro : Utilisez le thin provisioning dans Proxmox pour permettre une croissance flexible du disque sans pré-allouer la totalité de l'espace disque sur l'hôte.
