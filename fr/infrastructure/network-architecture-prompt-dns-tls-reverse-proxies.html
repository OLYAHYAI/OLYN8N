Architecture réseau : DNS, TLS et reverse proxies

Architecture réseau : DNS, TLS et reverse proxies

Objectif

Ce guide de référence présente des schémas d'architecture pour la mise en œuvre d'un accès réseau sécurisé dans des environnements homelab, en utilisant le DNS, les certificats TLS et les reverse proxies.

Schéma d'architecture : Services internes sécurisés

Couche 1 : Segmentation réseau

Internet → WAN → pfSense → LAN (Internal Services)
                     ↓
              Reverse Proxy (Nginx Proxy Manager)
                     ↓
         ┌───────────┼───────────┐
         ↓           ↓           ↓
    Proxmox        n8n      Databases
  (192.168.1.10) (.20)       (.30)

Couche 2 : Résolution DNS

Option A : Domaine public + Split DNS

Externe: proxmox.yourdomain.com → Public IP → Cloudflare Tunnel
Interne: proxmox.lab.local → 192.168.1.100 (Reverse Proxy)

Option B : DNS local uniquement

Tous les clients: proxmox.local → 192.168.1.100 (pfSense DNS Resolver)

Couche 3 : Terminaison TLS

Reverse Proxy (Nginx Proxy Manager)

Client (HTTPS) → Nginx Proxy Manager → Backend (HTTP/HTTPS)
              ↓
      Certificat Let's Encrypt ou auto-signé

Liste de vérification de l'implémentation

Configuration DNS

pfSense DNS Resolver :

Services → DNS Resolver → Host Overrides
- proxmox.lab → 192.168.1.100
- n8n.lab → 192.168.1.100
- db.lab → 192.168.1.100

Configuration du Reverse Proxy

Nginx Proxy Manager :

Proxy Hosts:
1. proxmox.lab → https://192.168.1.10:8006
2. n8n.lab → http://192.168.1.20:5678 (WebSockets: ON)
3. db.lab → http://192.168.1.30:5432

Options de certificat SSL

Option 1 : Let’s Encrypt (DNS Challenge) - Nécessite un domaine public - Utilise l'API Cloudflare - Renouvellement automatique tous les 90 jours

Option 2 : Auto-signé (mkcert) - Installation de l'AC locale requise - Aucune dépendance externe - Renouvellement manuel

Option 3 : Cloudflare Origin Certificate - Certificats gratuits de 15 ans - Uniquement pour les domaines proxifiés par Cloudflare - Reconnu par la périphérie Cloudflare (Cloudflare edge)

Couches de sécurité

1. Couche Réseau

  • Règles de pare-feu pfSense
  • Segmentation VLAN
  • Restrictions de ports

2. Couche Transport

  • Chiffrement TLS 1.3
  • Suites de chiffrement robustes
  • HTTP Strict Transport Security (HSTS)

3. Couche Application

  • Authentification (HTTP Basic Auth, OAuth)
  • Listes de contrôle d'accès
  • Limitation de débit (Rate limiting)

Schémas courants

Schéma 1 : Services de développement

Local DNS: service.lab
Certificate: Auto-signé (mkcert)
Access: LAN uniquement
Backend: HTTP

Schéma 2 : Services de production

Public DNS: service.yourdomain.com
Certificate: Let's Encrypt
Access: Internet (avec authentification)
Backend: HTTPS (validé)

Schéma 3 : Approche hybride

Interne: service.lab (mkcert, LAN)
Externe: service.yourdomain.com (Let's Encrypt, Cloudflare Tunnel)
Backend: Même service, chemins d'accès différents

Références des commandes rapides

Tester la résolution DNS

nslookup proxmox.lab
dig proxmox.lab +short

Tester le certificat TLS

openssl s_client -connect proxmox.lab:443 -servername proxmox.lab
curl -vI https://proxmox.lab

Vérifier la configuration du Proxy

curl -H "Host: proxmox.lab" http://192.168.1.100
curl -I https://proxmox.lab

Matrice de décision

Cas d'utilisation DNS Certificat Reverse Proxy
Dev/Tests Local (.lab) Auto-signé Optionnel
Prod interne Local (.lab) mkcert Requis
Accès externe Public (.com) Let’s Encrypt Requis
Données sensibles Local uniquement mkcert Requis + Auth

Meilleures pratiques

  1. Toujours utiliser HTTPS pour les interfaces web
  2. Mettre en œuvre une défense en profondeur (pare-feu + proxy + authentification)
  3. Utiliser des mots de passe robustes ou une authentification par clé
  4. Surveiller les journaux d'accès régulièrement
  5. Maintenir les certificats à jour (renouvellement automatique)
  6. Documenter votre architecture dans des diagrammes
  7. Tester les procédures de reprise après sinistre

Conclusion

Une architecture réseau appropriée intégrant le DNS, le TLS et les reverse proxies permet de créer une infrastructure homelab sécurisée et gérable. Choisissez les schémas en fonction de vos exigences de sécurité et de vos besoins d'accès.

Pour une implémentation détaillée, consultez les articles connexes sur Nginx Proxy Manager et la configuration pfSense.
