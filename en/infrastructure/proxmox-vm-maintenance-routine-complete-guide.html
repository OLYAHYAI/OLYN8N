Proxmox VM Maintenance Routine

Tags: #proxmox #virtualization #maintenance #automation

Quick Reference

Task Frequency Command
Check host storage Weekly df -h /
Check thin pool usage Weekly lvs pve/data
Review snapshots Weekly qm listsnapshot <VMID>
Clean old snapshots As needed qm delsnapshot <VMID> <name>
Backup verification Monthly Check /var/lib/vz/dump/

Storage Health Checks

Monitor Host Storage

# Check root filesystem usage (keep below 90%)
df -h /

# Target: Keep usage below 85-90%
# Alert level: 90%+
# Critical: 95%+

Monitor LVM Thin Pool

# Check thin pool data usage
lvs pve/data

# Target: Keep data% below 80%
# Alert level: 80%+
# Critical: 90%+

# Full view with all volumes
lvs -a | grep data

Storage Expansion (If Needed)

# Check available space in volume group
vgdisplay pve | grep -i free

# Extend root filesystem
lvextend -l +100%FREE /dev/pve/root
resize2fs /dev/pve/root

# Verify expansion
df -h /

Snapshot Management Strategy

⚠️ Golden Rule

Keep maximum 1-2 snapshots per VM at any time

Snapshot Types

1. Golden Snapshot (Keep One)

  • Created after major successful setup
  • Represents stable, working state
  • Update rarely (quarterly or after major changes)
# Create golden snapshot
qm snapshot <VMID> golden-working-state

# Update golden (delete old, create new)
qm delsnapshot <VMID> golden-working-state
qm snapshot <VMID> golden-working-state

2. Pre-Change Snapshot (Temporary)

  • Created before risky operations
  • Delete within 24-48 hours
  • Never keep more than one
# Before risky change
qm snapshot <VMID> pre-change-$(date +%Y%m%d)

# After successful change (DELETE IT!)
qm delsnapshot <VMID> pre-change-20251030

# If change failed (rollback then delete)
qm rollback <VMID> pre-change-20251030
qm delsnapshot <VMID> pre-change-20251030

List and Clean Snapshots

# List all snapshots for a VM
qm listsnapshot <VMID>

# Clean up old snapshots
qm delsnapshot <VMID> <snapshot-name>

# Example cleanup script
for snap in $(qm listsnapshot 100 | grep -v current | grep -v golden | awk '{print $2}'); do
    qm delsnapshot 100 $snap
done

VM Performance Optimization

Disk Configuration

# Optimal disk settings for VM 100
qm set 100 -scsi0 local-lvm:vm-100-disk-0,cache=writeback,iothread=1

# Cache modes (in order of speed):
# - writeback (fastest, recommended)
# - writethrough (balanced)
# - directsync (slowest, avoid unless needed)

Check VM Disk Performance Inside VM

# Inside the VM, check disk I/O
iostat -x 1 10

# Watch for:
# - %util > 80% = disk bottleneck
# - await > 50ms = slow responses
# - iowait% high = I/O problems

Memory and Swap Check (Inside VM)

# Check memory usage
free -h

# Monitor for swapping (should be 0)
vmstat 1

# Look at 'si' and 'so' columns - should be 0

LVM Thin Pool Protection

Enable Auto-Extension

Prevents thin pool from filling and crashing VMs.

# Backup config
cp /etc/lvm/lvm.conf /etc/lvm/lvm.conf.backup

# Enable auto-extend
sed -i 's/# thin_pool_autoextend_threshold = 100/thin_pool_autoextend_threshold = 80/' /etc/lvm/lvm.conf
sed -i 's/# thin_pool_autoextend_percent = 20/thin_pool_autoextend_percent = 20/' /etc/lvm/lvm.conf

# Verify settings
grep thin_pool_autoextend /etc/lvm/lvm.conf

Expected output:

thin_pool_autoextend_threshold = 80
thin_pool_autoextend_percent = 20

Backup Strategy

Why Backups > Snapshots

  • Snapshots are NOT backups
  • Snapshots live on same storage (single point of failure)
  • Backups can be restored anywhere
  • Backups survive hardware failure

Setup Automated Backups

Via Web UI:

  1. Datacenter → Backup → Add
  2. Configure:
    • Schedule: Daily at 2 AM
    • Selection mode: Include selected VMs
    • Keep: Daily: 3, Weekly: 2, Monthly: 1
    • Compression: ZSTD
    • Mode: Snapshot

Via CLI:

# Manual backup
vzdump <VMID> --mode snapshot --compress zstd --storage local

# Backup to external storage (recommended)
vzdump <VMID> --storage backup-storage --compress zstd

Backup Retention Strategy

Daily:   Keep 3 (today, yesterday, 2 days ago)
Weekly:  Keep 2 (last 2 weeks)
Monthly: Keep 1 (last month)

Emergency Troubleshooting

VM Freezing Checklist

1. Check Host Storage (Most Common)

df -h /
# If > 90%, expand or clean up immediately

2. Check Thin Pool

lvs pve/data
# If data% > 80%, delete old snapshots

3. Check VM Disk Cache

qm config <VMID> | grep cache
# Should be: cache=writeback

4. Inside VM: Check I/O

iostat -x 1
# Look for high %util or await times

5. Inside VM: Check Memory

free -h
vmstat 1
# Check for swapping (si/so columns)

Quick Cleanup Commands

# Host cleanup
apt clean
apt autoclean
journalctl --vacuum-time=7d
rm -rf /var/cache/apt/archives/*

# Find space hogs
du -h / --max-depth=3 2>/dev/null | sort -h | tail -20

# Remove old kernels (keep current + 1)
dpkg --list | grep linux-image
apt remove linux-image-X.X.X-X-pve

Automation Scripts

Weekly Health Check Script

#!/bin/bash
# Save as: /root/proxmox-health-check.sh

echo "=== Proxmox Health Check - $(date) ===" | tee -a /var/log/proxmox-health.log

# Check root storage
echo "Root Storage:" | tee -a /var/log/proxmox-health.log
df -h / | grep mapper | tee -a /var/log/proxmox-health.log

# Check thin pool
echo "Thin Pool Usage:" | tee -a /var/log/proxmox-health.log
lvs pve/data | tee -a /var/log/proxmox-health.log

# List snapshots for each VM
for vmid in $(qm list | awk 'NR>1 {print $1}'); do
    echo "VM $vmid snapshots:" | tee -a /var/log/proxmox-health.log
    qm listsnapshot $vmid | tee -a /var/log/proxmox-health.log
done

# Alert if storage > 85%
USAGE=$(df / | awk 'NR==2 {print $5}' | sed 's/%//')
if [ $USAGE -gt 85 ]; then
    echo "⚠️  WARNING: Root storage at ${USAGE}%" | tee -a /var/log/proxmox-health.log
fi

echo "===================================" | tee -a /var/log/proxmox-health.log

Make it executable and schedule

chmod +x /root/proxmox-health-check.sh

# Add to crontab (every Sunday at 8 AM)
crontab -e
# Add this line:
0 8 * * 0 /root/proxmox-health-check.sh

Monitoring Thresholds

Set Alerts When:

Metric Warning Critical Action
Root Storage 85% 95% Expand or clean
Thin Pool Data 75% 85% Delete snapshots
Thin Pool Meta 75% 85% Alert admin
VM Disk I/O Wait 20% 40% Check cache mode
Snapshot Count 3 per VM 5 per VM Clean immediately

Lessons Learned

Root Cause of Previous Freezing

  1. ✅ 100% full root filesystem - primary cause
  2. ✅ Cache mode: directsync - slow I/O
  3. ✅ Snapshot chain - space inefficiency
  4. ✅ No thin pool auto-extend - risk of filling

Solutions Applied

  1. ✅ Expanded root from 96GB → 112GB
  2. ✅ Changed cache: directsync → writeback
  3. ✅ Deleted unnecessary snapshot chain
  4. ✅ Enabled thin pool auto-extension
  5. ✅ Established snapshot management strategy

Prevention Strategy

  • Monitor storage weekly
  • Never exceed 2 snapshots per VM
  • Use backups, not snapshots, for safety
  • Enable alerts at 85% storage usage
